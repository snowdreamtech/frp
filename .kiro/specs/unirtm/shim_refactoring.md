# UniRTM Shim 架构重构实施计划 (已完成)

## 1. 核心目标

标准化 UniRTM Shim 架构，实现扁平化、高性能的单二进制代理机制，并提升跨平台一致性。

## 2. 已完成的工作

### A. 扁平化与代理架构

- **单一入口代理**：
    - **Unix (macOS/Linux)**：将 Shim 修改为指向 `unirtm` 的软链接。
    - **Windows**：将 Shim 修改为指向 `unirtm` 的**硬链接** (Hard Link, Scoop 模式变体)。这提供了原生的 EXE 执行性能且不占用额外磁盘空间，同时避免了创建软链接所需的管理员权限。
- **目录扁平化**：强制所有 Shim 文件生成在 `shims/` 根目录下，避免嵌套目录导致的 PATH 复杂性。
- **动态分发**：在 `unirtm shim` 模式下，通过读取调用名称（`os.Args[0]` 或显式参数）动态解析并执行对应的工具二进制。

### B. 健壮的可执行文件解析 (`ResolveExecutable`)

- **候选集匹配策略**：从数据库加载所有安装项，并根据调用名称（如 `node`）智能筛选匹配的工具（支持 `node` 匹配 `node` 或 `node-25.9.0`）。
- **多路径扫描**：Provider 现在会同时扫描安装根目录和 `bin/` 子目录，显著提升了对 GitHub 来源工具（布局不标准）的兼容性。
- **优先级排序**：支持根据环境变量（如 `UNIRTM_NODE_VERSION`）或当前项目配置优先选择版本。

### C. 垃圾 Shim 过滤

- **严格的可执行性检查**：改进了 `GenericProvider.isExecutable`，排除了 `.md`, `.py`, `.json`, `.txt` 等常见非二进制文件，即使它们具有执行权限。
- **优选机制 (`pickBestExecutables`)**：在生成 Shim 时对搜索到的文件进行评分。已优化为**包容性策略**：自动保留 `bin/` 目录下的所有有效二进制文件（如 `java` 及其配套的 `javac`），并放宽评分区间，确保“全家桶”类工具的关联程序也能正确生成 Shim。

### D. 系统稳定性与验证

- **数据库一致性**：确认了 Shim 进程能够正确读取 SQLite 数据库。
- **清理 with 重置**：通过手动清理旧 Shim 并重新执行 `reshim`，验证了新机制的有效性。

## 3. 下一步建议

1. **Windows 性能优化**：虽然目前使用 Wrapper 脚本，但可以考虑在 Windows 上也提供类似 `mise` 的高性能二进制 Shim 替代方案.
